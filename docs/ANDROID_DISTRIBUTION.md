# Android Distribution

Personal notes for publishing the Android APK without going through the Play
Store just yet.

## Main channel

For now, the simplest approach is to publish a universal APK in GitHub releases:

- Git tag in `v2.0.4` format;
- public release, not a draft;
- attached file named `avocook.apk`.

The name is intentionally simple. Users don't have to choose a CPU architecture.

## Before a release

1. Update `expo.version` in `app.json`.
2. Run local checks:

   ```bash
   npm run typecheck
   npm test
   npm run lint
   ```

3. Generate a signed Android build.
4. Create the Git tag and the GitHub release.
5. Attach `avocook.apk`.
6. Install the previous version on a phone and verify that the update banner
   detects the new release.

The update checker compares the app's Expo version against the latest GitHub
tag.

## IzzyOnDroid

IzzyOnDroid is the first Android repository to target, because it accepts APKs
published in GitHub Releases.

Handy info for a submission request:

- name: AvoCook;
- package: `app.avocook.mobile`;
- repository: `https://github.com/Logarex/AvoCook`;
- releases: `https://github.com/Logarex/AvoCook/releases`;
- licence: MIT;
- APK file: `avocook.apk`.

Short description:

```text
AvoCook is a recipe book for Android and iOS. It works offline and can also
sync with a Nextcloud Cookbook instance.
```

Key points:

- no ads;
- no analytics;
- no account required in local mode;
- no intentional dependency on Google Play Services;
- sync only with the Nextcloud server chosen by the user.

If the universal APK is considered too large, generate an `arm64-v8a` APK as
well for IzzyOnDroid. Keep `avocook.apk` for public releases anyway.

## F-Droid

F-Droid will be more involved, because that repository rebuilds apps from
source. The native Android project generated by Expo will likely need to be
stabilised before making a serious submission.

Probable starting command:

```bash
npm ci
npx expo prebuild --platform android --clean
cd android
./gradlew assembleRelease
```

Don't commit to F-Droid before verifying that this build is reproducible in a
clean environment.

## APK size

The universal APK is large because it bundles native libraries for multiple
architectures. To inspect an APK:

```bash
unzip -l avocook.apk | sort -nr -k1 | head -40
```

Folders to look at:

```text
lib/arm64-v8a/
lib/armeabi-v7a/
lib/x86_64/
```

ABI splits are configured by `plugins/withAndroidSplitsAndOptimizations.js`.
I prefer not to expose these variants to users as long as the direct GitHub
download remains the main channel.
